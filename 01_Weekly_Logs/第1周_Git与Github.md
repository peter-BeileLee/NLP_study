## 1. Git 是什么？

* **通俗解释**：**单机版的“代码时光机”和“后悔药”。**
* **大白话含义**：它是一个装在你电脑本地的软件。每当你写完一段代码，就可以用 Git 存一个“存档点”。如果以后代码写崩了，或者不小心删错了，你可以一键让代码“瞬间穿越”回过去的任何一个存档点。它还能记录是谁、在什么时候、改了哪一行代码。

---

## 2. GitHub 是什么？

* **通俗解释**：**云端版的“代码朋友圈”和“大型同性交友/托管网站”。**
* **大白话含义**：Git 是在你本地电脑运行的，而 GitHub 是一个网站。你把自己电脑上的 Git 存档点推送到 GitHub 网站上，这就叫“代码开源”或“云端备份”。
* **区别**：**Git 是工具（单机版），GitHub 是平台（联网版）。**

---

## 3. commit 是什么？

* **通俗解释**：**玩游戏时的“保存历史存档”。**
* **大白话含义**：当你写完一个功能（比如做好了登录页面），你在命令行输入 `git commit -m "完成了登录功能"`，这就意味着你正式拍了一张“代码快照”并存入了时光机。以后这就是一个合法的历史版本，随时可以回退到这里。

---

## 4. README 是什么？

* **通俗解释**：**项目的“产品说明书”或“门面担当”。**
* **大白话含义**：通常在项目根目录下有一个 `README.md` 文件。当别人（或面试官）在 GitHub 上点开你的项目时，第一眼看到的就是这个文件渲染出来的网页。里面用文字和图片写着：这个项目是干嘛的、怎么安装、怎么运行。

---

## 5. .gitignore 是什么？

* **通俗解释**：**Git 时光机的“免打扰名单”。**
* **大白话含义**：这是一个隐藏的文本文件。里面写了什么，Git 就会彻底无视什么。
* **为什么要它**：比如咱们前面刚创建的 `env` 虚拟环境文件夹，里面有几万个无聊的系统文件，还有一些缓存垃圾，我们不希望把这些垃圾也存进 Git 存档、上传到 GitHub。我们就可以在 `.gitignore` 里面写上一行 `env/`，Git 就会自动假装看不见它，只专注备份你写的核心代码。

---

## 6. 为什么不能把密码和 API key 上传到 GitHub？

* **通俗解释**：**相当于把自家的防盗门钥匙挂在市中心广场的大屏幕上。**
* **大白话含义**：
1. **人人可见**：GitHub 默认是公开的。全天候有无数的“黑客机器人”在 GitHub 上疯狂扫描 `password`、`API_KEY`（比如 OpenAI 的付费密钥、数据库密码）。
2. **一秒被盗**：只要你一不小心把包含密钥的代码 push 上去，几秒钟内黑客的脚本就能扫到它，然后拿去疯狂刷量消费，或者黑进你的服务器删库勒索。
3. **历史难删**：因为 Git 记录的是“历史存档（commit）”。就算你发现错了，赶紧在代码里删掉密码重新上传，黑客依然可以去查你上一次的 commit 记录，把密码翻出来。所以敏感信息必须放在本地的 `.env` 文件里，并用 `.gitignore` 锁死！
# requirements.txt 核心速查手册

## 一、 什么是 requirements.txt？

* **定义**：Python 官方（pip）标准的纯文本环境配置文件。
* **作用**：精确记录当前项目所依赖的**第三方 Python 库名称及其版本号**。

---

### 二、 核心操作命令

#### 1. 导出清单（在开发电脑上打包环境）

将当前虚拟环境中所有已安装的 Python 包和版本号自动抓取并写入文件：

```cmd
pip freeze > requirements.txt

```

#### 2. 导入安装（在部署/新电脑上一键还原）

让 Python 自动读取文件中的每一行配置，批量下载并安装完全相同的依赖库：

```cmd
pip install -r requirements.txt

```

---

### 三、 文件内容示例

文件内部为极为干净的纯文本格式：

```text
numpy==1.26.4
pandas==2.2.2
torch==2.3.0

```

---

### 四、 核心价值

* **消除环境差异**：彻底解决“在我的电脑上能跑，在别人电脑上跑不起来”的兼容性难题。
* **轻量化共享**：不需要将几万个、几个G的本地环境文件打包发给别人，只需共享这一份几 KB 的文本“图纸”即可。
